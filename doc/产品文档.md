# AhaAgent 产品设计文档（修订版）

## 1. 产品概述

AhaAgent 是一个通过命令行启动、在本地 Web 页面交互的 AI Coding Agent。它的目标是帮助用户完成开发任务分解、代码改动、验证与结果汇报。

本修订版目标是让方案在“可用”的同时满足“可控、安全、可恢复、可审计”。

## 2. V1 范围与非目标

### 2.1 V1 范围（必须实现）

- 本地 Assistant UI（聊天主流程 + 任务状态 + 审批弹窗 + 可选开发者控制台）。
- 标准工具链：`read_file`、`list_dir`、`grep`、`diff_edit`、`write_file`、`run_command`。
- 安全基线：工作区沙箱、敏感数据防泄漏、写操作审批、协议鉴权。
- 任务编排：子任务状态、断点恢复、取消与重试控制。
- 扩展能力：支持 MCP，但仅在“已安装且已授权”的扩展内运行。

### 2.2 V1 非目标（暂不做）

- 模型自主联网下载并安装第三方扩展。
- 把 `vm` 当作安全边界执行不可信插件。
- 无约束地把本地读取内容自动发送到外部 LLM。

## 3. 设计原则

1.  **默认拒绝（Deny by Default）**：权限、扩展、外发数据全部默认拒绝。
2.  **显式授权（Explicit Approval）**：高风险动作必须用户确认。
3.  **最小权限（Least Privilege）**：每个工具、扩展、任务只拿最少权限。
4.  **幂等可恢复（Idempotent & Recoverable）**：断线、重试、重复点击都不能导致重复破坏。
5.  **可审计（Auditable）**：关键决策、审批、执行结果可追溯。

## 4. 核心能力设计（修订后）

### 4.1 助理式交互界面

- 主聊天流仅展示自然语言结果与高层进度。
- 底层日志放入“开发者控制台”抽屉，不默认打扰用户。
- 启动 CLI 后自动打开 `http://localhost:<port>`，前端通过 WebSocket 接收流式状态。

### 4.2 任务编排与状态可视化

- 复杂任务拆分为任务树/DAG，UI 仅展示必要状态：`pending/running/blocked/success/failed/cancelled`。
- 对“等待审批”状态必须强提示，避免用户误判为卡死。
- 每个任务必须有唯一 `taskId` 与 `requestId`，用于恢复与排错。

### 4.3 编码与文件修改能力

- 默认通过 `diff_edit` 做精确改动，不允许模型整文件重写大文件。
- 改动前必须读取最新文件版本；改动时进行版本校验，不匹配则回退到“重新读取并重规划”。
- 所有写入能力进入变更队列，避免并发写穿透。

### 4.4 多模型供应商支持

- V1 技术栈统一为 **Node.js + TypeScript**。
- 使用 OpenAI-compatible 适配层管理不同模型供应商（OpenAI/Claude/DeepSeek 等）。
- 模型配置来自本地 `~/.aha/config.json` 与环境变量，支持会话级切换。

### 4.5 记忆体系（带安全约束）

- **短期记忆**：当前会话上下文，任务结束后总结压缩。
- **长期记忆**：本地持久化（SQLite），记录偏好与项目事实，包含 `access_count/last_accessed_at` 做衰减淘汰。
- **技能沉淀**：本地 `~/.aha/skills/*.md`，但执行前仍需经过权限与策略检查。
- **新增约束**：记忆写入与召回都要经过敏感信息扫描，禁止把密钥、令牌、私钥落入记忆库。

### 4.6 外部技能生态（MCP）

- AhaAgent 可作为 MCP Client 接入外部生态。
- 外部技能与本地技能在“路由接口”层统一，但在“安全等级”上不等同：
  - 本地技能：受本地策略约束。
  - 外部技能：必须完成安装校验、权限声明、用户授权后才可启用。

### 4.7 扩展安装与热加载（高风险重构后）

这是本次修订的重点，替代“模型可直接安装并执行”的方案。

- **模型只能建议，不能直接安装**：
  - LLM 可调用 `propose_install_skill(name/source)` 提出建议。
  - 真正安装动作只能由用户在 UI 或 CLI 明确触发。
- **安装信任链**：
  1.  来源白名单（官方 registry/可信 Git 仓库）。
  2.  校验 `manifest + SHA256 + 签名`。
  3.  下载到隔离暂存区，先扫描再安装。
  4.  解析权限声明（文件、网络、命令执行、可访问目录）。
  5.  用户二次确认后启用。
- **热加载策略**：
  - 热加载只刷新“扩展元数据/工具目录”。
  - 不可信代码在隔离运行时中启动，不在主进程 `require()` 执行。
  - 新扩展启动失败自动回滚到上一个稳定快照。

### 4.8 安全与权限沙箱

#### (1) 工作区沙箱（Path Jail 2.0）

- 仅允许访问启动时声明的 `allowed_workspaces`。
- 路径校验使用 `realpath` 与工作区真实路径比对，而非仅靠字符串前缀。
- 拒绝符号链接逃逸、路径穿越、相对路径越权。
- 文件打开后再次做句柄级校验，降低 TOCTOU 风险。

#### (2) 敏感数据防泄漏（Data Egress Guard）

- “读操作”不再被视为天然安全。
- 增加数据分级：
  - `public`：可直接用于模型上下文。
  - `restricted`：需脱敏后可用。
  - `secret`：禁止发送到外部模型。
- 默认黑名单：`.env*`, `*.pem`, `*.key`, `id_rsa*`, `secrets.*`, `.npmrc`, `.ssh/*`。
- 日志与记忆写入同样走脱敏流程。

#### (3) 高危动作审批与限权

- `write_file/delete_file/run_command/install_extension` 必须审批。
- 审批包含：目标、diff 预览、风险级别、权限范围、有效期。
- 允许“本任务临时放行”，但必须带作用域和过期时间，任务结束自动失效。

### 4.9 健壮性与可恢复性

- **防重试风暴**：每步 `max_retries`，超过即阻断并请求人工决策。
- **并发一致性**：
  - 读可并发；写串行。
  - 文件级锁 + 版本号校验，冲突即回滚重规划。
- **审批断点恢复**：审批流程与 LLM 长请求解耦，审批后以新请求恢复。
- **取消语义**：`cancel_task` 必须定义状态迁移与资源释放（锁、子进程、队列项）。
- **日志可观测性**：info/error 分流、结构化日志、按字段脱敏。

## 5. 关键风险闭环（对照）

1.  **扩展供应链与 RCE 风险**：通过“模型不可直接安装 + 签名校验 + 隔离运行”闭环。
2.  **沙箱绕过风险**：通过“realpath + 句柄复检 + 工作区白名单 + 默认拒绝”闭环。
3.  **本地端口被恶意网页调用风险**：通过“会话密钥、Origin 校验、审批 nonce”闭环。
4.  **重复执行与断线恢复风险**：通过“requestId/idempotencyKey/checkpoint”闭环。
5.  **日志与记忆泄密风险**：通过“全链路脱敏 + 分级外发策略”闭环。

## 6. 技术栈与版本策略

- **后端**：Node.js + TypeScript（统一口径，不做 Python 双栈）。
- **前端**：Vue 3 或 React（二选一），WebSocket 实时通信。
- **存储**：SQLite（任务、检查点、记忆、审计日志索引）。
- **扩展协议**：MCP Client + 本地扩展运行时。
